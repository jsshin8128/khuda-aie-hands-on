# 프롬프트 엔지니어링 가이드

> 이 가이드는 프롬프트를 처음 체계적으로 배우는 분들을 위해 작성되었습니다.  
> "왜 프롬프트가 중요한가?"부터 시작해, 핵심 개념을 하나씩 쌓아 가며 우리 프로젝트 코드와 연결합니다.

---

## 0. 이 프로젝트에서 프롬프트가 왜 중요한가

우리가 만든 서비스를 떠올려 봅시다. 블로그 글 URL을 받아서 요약을 돌려주는 서비스입니다. 핵심은 이 줄입니다.

```python
# body: SummaryRequest — FastAPI가 HTTP 요청 바디를 파싱해 Pydantic 객체 형태로 body에 넘긴다.
# SummaryRequest는 url, output_format 두 필드를 가진다.

# body.url로 페이지를 크롤링해 제목과 본문을 추출한다.
content_text, title = await fetch_url(body.url)

chain = _PROMPT | _LLM
# chain.ainvoke() — 여기서 실제 LLM(GPT-4o-mini) 호출이 일어난다.

# _PROMPT의 {title}, {content_text} 자리에 값을 채운 최종 프롬프트를 _LLM에 전달한다.
result = await chain.ainvoke({"title": title or "", "content_text": content_text})
```

`_LLM`은 GPT-4o-mini입니다. 같은 글을 넣어도, `_PROMPT`가 무엇이냐에 따라 결과가 완전히 달라집니다.

프롬프트가 이렇다면:

```
블로그 글을 요약해 줘.
제목: {title}
내용: {content_text}
```

LLM은 자유롭게 응답합니다. HTTP 응답 바디가 json 형태가 아니기 때문에, `json.loads(result.content)` 에서 에러가 납니다.

```python
# result.content 예시 — 모델이 자유롭게 응답한 경우
result.content = """
이 블로그 글은 파이썬의 비동기 처리를 다룹니다.
핵심 내용은 asyncio, 코루틴, 이벤트 루프입니다.
초보자도 이해하기 쉽게 설명되어 있습니다.
"""

json.loads(result.content)
# JSONDecodeError: Expecting value: line 1 column 1 (char 0)
```

하지만 프롬프트가 이렇다면:

```
블로그 글을 읽고 아래 JSON 형식으로만 응답하세요. 다른 텍스트 없이 순수 JSON만 반환하세요.

{
  "summary": "2~3문장 핵심 요약",
  "key_points": ["포인트 1", "포인트 2", "포인트 3"]
}

제목: {title}
내용: {content_text}
```

```python
# result.content 예시 — 모델이 JSON 형식으로 응답한 경우
result.content = """
{
  "summary": "이 블로그 글은 파이썬의 비동기 처리를 다룹니다. asyncio와 코루틴의 동작 원리를 설명하며, 이벤트 루프가 어떻게 작업을 스케줄링하는지 보여줍니다.",
  "key_points": ["asyncio는 단일 스레드에서 동시성을 구현한다", "코루틴은 await 지점에서 실행을 양보한다", "이벤트 루프가 준비된 코루틴을 순서대로 실행한다"]
}
"""

data = json.loads(result.content)
# 파이썬이 이해할 수 있는 자료구조 dict 형태로 변환 성공!
```

이제 파싱이 안정적으로 됩니다. 모델 코드는 한 줄도 안 바꿨는데, 프롬프트 하나로 서비스의 안정성이 결정됩니다.

이게 프롬프트 엔지니어링이 필요한 첫 번째 이유입니다.

# 1. 프롬프트 이해하기

## 프롬프트의 세 가지 구성 요소

잘 만든 프롬프트는 보통 세 가지 요소로 구성됩니다.

### 작업 설명 (Instruction)

모델이 **무엇을** 해야 하는지, **어떤 형식**으로 답해야 하는지를 알려줍니다. 역할을 부여할 수도 있습니다.

```
블로그 글을 읽고 아래 JSON 형식으로만 응답하세요.
다른 텍스트 없이 순수 JSON만 반환하세요.

{
  "summary": "2~3문장 핵심 요약",
  "key_points": ["포인트 1", "포인트 2", "포인트 3"]
}
```

### 예시 (Few-shot Examples)

모델이 기대하는 입출력이 어떤 모습인지 보여줍니다. 작업 설명만으로 부족할 때 예시가 큰 역할을 합니다.

```
제목: "파이썬 asyncio 완전 정복"
내용: asyncio와 코루틴을 이용해 단일 스레드에서 동시성을 구현하는 방법을 설명합니다. ...

{
  "summary": "파이썬 asyncio를 사용해 단일 스레드에서 동시성을 구현하는 방법을 다룹니다. 코루틴과 이벤트 루프의 동작 원리를 실제 코드 예시와 함께 설명합니다.",
  "key_points": ["asyncio는 단일 스레드에서 동시성을 구현한다", "코루틴은 await 지점에서 실행을 양보한다", "이벤트 루프가 준비된 코루틴을 순서대로 실행한다"]
}
```

예시를 몇 개 주느냐에 따라 Zero-shot, One-shot, Few-shot이라고 부릅니다. 뒤에서 다룰게요!

### 실제 작업 (Task)

모델이 실제로 처리해야 할 입력입니다. 우리 프로젝트에서는 `{title}`과 `{content_text}`가 여기에 해당합니다.

```
제목: {title}
내용: {content_text}
```

## 세 요소를 합치면

우리 프로젝트의 블로그 요약 프롬프트를 예로 들면 이렇게 됩니다.

```
[작업 설명]
블로그 글을 읽고 아래 JSON 형식으로만 응답하세요.
다른 텍스트 없이 순수 JSON만 반환하세요.

{
  "summary": "2~3문장 핵심 요약",
  "key_points": ["포인트 1", "포인트 2", "포인트 3"]
}

[예시]
제목: "파이썬 asyncio 완전 정복"
내용: asyncio와 코루틴을 이용해 단일 스레드에서 동시성을 구현하는 방법을 설명합니다. ...

{
  "summary": "파이썬 asyncio를 사용해 단일 스레드에서 동시성을 구현하는 방법을 다룹니다. 코루틴과 이벤트 루프의 동작 원리를 실제 코드 예시와 함께 설명합니다.",
  "key_points": ["asyncio는 단일 스레드에서 동시성을 구현한다", "코루틴은 await 지점에서 실행을 양보한다", "이벤트 루프가 준비된 코루틴을 순서대로 실행한다"]
}

[실제 작업]
제목: {title}
내용: {content_text}
```

세 요소가 항상 다 필요한 건 아닙니다. 간단한 작업은 작업 설명만으로 충분하고, 복잡하거나 형식이 중요한 작업일수록 예시가 중요해집니다.

## 인컨텍스트 학습: 제로샷과 퓨샷

프롬프트를 통해 모델에게 무엇을 해야 할지 가르치는 것을 **인컨텍스트 학습(in-context learning)** 이라고 합니다.

원래 모델은 학습할 때만 새 행동을 배울 수 있었습니다. 그런데 2020년 [GPT-3 논문](https://arxiv.org/abs/2005.14165)이 이걸 뒤집었어요. **가중치 업데이트 없이 프롬프트 안의 예시만으로도 모델이 원하는 행동을 한다**는 걸 보여줬거든요.

덕분에 모델이 학습 이후의 변화에도 대응할 수 있게 됐습니다. 오래된 문서로 학습된 모델이라도, 컨텍스트에 최신 변경사항을 넣어주면 그걸 바탕으로 답할 수 있게됐어요! 재학습 없이도요.

### 제로샷 vs 퓨샷

프롬프트에 제공하는 각 예시를 **샷(shot)** 이라고 부릅니다.

```
# 제로샷 (0-shot): 예시 없음
블로그 글을 읽고 JSON 형식으로 요약하세요.
제목: "도커 컨테이너 완전 정복"
내용: 도커의 기본 개념부터 실전 배포까지 다룹니다...
요약:

# 원샷 (1-shot): 예시 1개
블로그 글을 읽고 JSON 형식으로 요약하세요.

제목: "파이썬 asyncio 완전 정복"
내용: asyncio와 코루틴을 이용해 단일 스레드에서 동시성을 구현하는 방법을 설명합니다...
→ {"summary": "파이썬 asyncio로 단일 스레드 동시성을 구현하는 방법을 다룹니다.", "key_points": ["asyncio는 단일 스레드에서 동시성을 구현한다", "코루틴은 await 지점에서 실행을 양보한다", "이벤트 루프가 준비된 코루틴을 순서대로 실행한다"]}

제목: "도커 컨테이너 완전 정복"
내용: 도커의 기본 개념부터 실전 배포까지 다룹니다...
요약:

# 퓨샷 (few-shot): 예시 여러 개
블로그 글을 읽고 JSON 형식으로 요약하세요.

제목: "파이썬 asyncio 완전 정복"
내용: asyncio와 코루틴을 이용해 단일 스레드에서 동시성을 구현하는 방법을 설명합니다...
→ {"summary": "파이썬 asyncio로 단일 스레드 동시성을 구현하는 방법을 다룹니다.", "key_points": ["asyncio는 단일 스레드에서 동시성을 구현한다", "코루틴은 await 지점에서 실행을 양보한다", "이벤트 루프가 준비된 코루틴을 순서대로 실행한다"]}

제목: "Git 브랜치 전략 총정리"
내용: Git Flow, GitHub Flow, Trunk-based Development를 비교하며 팀에 맞는 전략을 선택하는 방법을 설명합니다...
→ {"summary": "대표적인 Git 브랜치 전략 세 가지를 비교하고 팀 규모와 배포 주기에 따라 적합한 전략을 고르는 기준을 제시합니다.", "key_points": ["Git Flow는 정기 릴리즈 팀에 적합하다", "GitHub Flow는 지속적 배포 환경에 맞다", "Trunk-based는 소규모 고숙련 팀에 유리하다"]}

제목: "도커 컨테이너 완전 정복"
내용: 도커의 기본 개념부터 실전 배포까지 다룹니다...
요약:
```

정확히 몇 개가 필요한지는 모델과 작업에 따라 다릅니다. 일반적으로 예시가 많을수록 성능이 좋아지지만, **모델의 최대 컨텍스트 길이**와 **추론 비용**에 따라 조절이 필요합니다.

### 강한 모델일수록 퓨샷의 이점이 줄어든다

GPT-3에서는 퓨샷이 제로샷에 비해 성능 향상이 뚜렷했습니다. 하지만 GPT-4 같은 강력한 모델에서는 퓨샷이 제로샷 대비 제한적인 개선만을 가져오는 경우도 있습니다. 모델이 강해질수록 지시를 이해하는 능력 자체가 올라가기 때문입니다.

단, 이 결과가 항상 맞는 건 아닙니다. 모델이 학습 데이터에서 거의 보지 못한 도메인(예: 사내 용어)이라면, 예시 몇 개가 성능을 크게 바꿀 수 있습니다.

---

그런데 여기서 자연스럽게 이런 생각이 들 수 있어요. 예시를 잘 골랐는데, 프롬프트를 조금만 바꿨더니 갑자기 결과가 달라졌다면요? "JSON으로 응답해"라고 썼을 때랑 "json으로 응답해"라고 썼을 때 결과가 다르다면 안되겠죠? 이게 바로 **강건성** 문제예요.

## 모델 강건성 (Robustness)

프롬프트를 조금 바꿨을 때 결과가 얼마나 달라지는지를 **강건성**이라고 합니다.

예를 들어, 숫자 `5`를 문자 `five`로 바꾸면 어떻게 될까요?

강건성이 낮은 모델은 이런 변화에도 출력이 크게 달라집니다. 강건성이 높은 모델은 표현이 조금 달라져도 일관된 결과를 냅니다.

```
# 강건성이 낮은 모델
"JSON 형식으로 응답하세요" → { "summary": "..." }  ✅
"json 형식으로 응답하세요" → ```json { "summary": "..." } ```  ❌

# 강건성이 높은 모델
둘 다 같은 결과를 냄 ✅
```

**모델이 강력해질수록 강건성도 높아집니다.** GPT-4o가 GPT-3.5보다 강건한 이유입니다. 더 강한 모델을 쓰면 프롬프트를 정확하게 짜는 데 드는 시행착오가 줄어듭니다.

강건성은 프롬프트를 조금씩 변형해서 결과가 얼마나 바뀌는지 실험해보는 방식으로 측정할 수 있습니다.

---

그럼 강건성을 높이려면 어떻게 해야 할까요? 사실 우리 프로젝트 코드를 다시 보면, 이미 좋은 방법을 쓰고 있어요. 작업 설명과 실제 입력을 아예 **별도 역할로 분리**해두는 거예요. 그게 바로 `"system"`과 `"user"`의 구분입니다.

## 시스템 프롬프트와 사용자 프롬프트

우리 프로젝트의 `_PROMPT`를 다시 봅시다.

```python
_PROMPT = ChatPromptTemplate.from_messages([
    ("system", "블로그 글을 읽고 아래 JSON 형식으로만 응답하세요..."),
    ("user", "제목: {title}\n\n내용:\n{content_text}"),
])
```

`"system"`과 `"user"`가 보이죠? 이게 바로 **시스템 프롬프트**와 **사용자 프롬프트**의 구분입니다.

### 각각의 역할


| 구분           | 역할                       | 누가 작성하나      |
| ------------ | ------------------------ | ------------ |
| **시스템 프롬프트** | 작업 설명 — 모델이 어떻게 행동해야 하는지 | 애플리케이션 개발자   |
| **사용자 프롬프트** | 실제 작업 — 지금 처리해야 할 입력     | 사용자 (또는 개발자) |


우리 프로젝트를 예로 들면 이런 구조가 됩니다.

```
[시스템 프롬프트]
블로그 글을 읽고 아래 JSON 형식으로만 응답하세요.
다른 텍스트 없이 순수 JSON만 반환하세요.

{
  "summary": "2~3문장 핵심 요약",
  "key_points": ["포인트 1", "포인트 2", "포인트 3"]
}

[사용자 프롬프트]
제목: 파이썬 asyncio 완전 정복
내용: asyncio와 코루틴을 이용해 단일 스레드에서 동시성을 구현하는 방법을 설명합니다...
```

ChatGPT를 포함한 거의 모든 생성형 AI 애플리케이션에는 시스템 프롬프트가 있습니다. 사용자는 시스템 프롬프트를 볼 수 없지만, 그게 애플리케이션의 성격과 행동 방식을 결정합니다.

### 채팅 템플릿: 모델 내부에서 어떻게 합쳐지나

시스템 프롬프트와 사용자 프롬프트가 분리되어 있어 보여도, 모델은 이 둘을 **하나의 최종 프롬프트로 합쳐서** 처리합니다. 모델마다 합치는 방식, 즉 **채팅 템플릿**이 다릅니다.

라마 2의 채팅 템플릿은 이렇게 생겼습니다.

```
<s>[INST] <<SYS>>
{{ 시스템_프롬프트 }}
<</SYS>>
{{ 사용자_메시지 }} [/INST]
```

라마 3에서는 이렇게 바뀌었습니다.

```
<|begin_of_text|><|start_header_id|>system<|end_header_id|>
{{ system_prompt }}<|eot_id|><|start_header_id|>user<|end_header_id|>
{{ user_message }}<|eot_id|><|start_header_id|>assistant<|end_header_id|>
```

같은 메타가 만든 모델이지만 버전이 바뀌면서 템플릿이 달라졌습니다.

그래서 우리 프로젝트는 **LangChain**을 씁니다. LangChain이 모델에 맞는 채팅 템플릿을 자동으로 처리해줘서, 이 부분을 신경 쓰지 않아도 돼요!
추상화의 이점 덕에, 우리는 `"system"`/`"user"` 구분만 해주면 됩니다.

```python
# 우리가 작성하는 것 — 채팅 템플릿은 신경 안 써도 됨 (채팅 템플릿은 모델 개발자가 정의한 것!)
_PROMPT = ChatPromptTemplate.from_messages([
    ("system", "블로그 글을 읽고 아래 JSON 형식으로만 응답하세요..."),
    ("user", "제목: {title}\n\n내용:\n{content_text}"),
])

# LangChain이 내부적으로 모델에 맞는 템플릿으로 변환해서 전달
chain = _PROMPT | _LLM
```

만약 LangChain 없이 모델 API를 직접 호출한다면, 모델 문서에서 채팅 템플릿을 직접 확인하고 올바르게 적용해야 합니다.

### 잘못된 채팅 템플릿의 위험

채팅 템플릿이 맞지 않으면 성능 문제가 생깁니다. 그런데 우리는 랭체인으로 추상화를 했었죠? 문제는 **템플릿이 틀려도 모델이 그럴듯한 응답을 내놓기 때문에 문제가 겉으로 잘 드러나지 않는다**는 것입니다.

따라서 디버깅을 통해 `chain.invoke()` 전에 완성된 프롬프트를 먼저 찍어보는 습관이 중요해요. (실습에서 해볼게요!)

```python
# chain에 넣기 전에 완성된 프롬프트 확인
prompt_value = _PROMPT.invoke({"title": title, "content_text": content_text})
print(prompt_value.to_string())
```

---

그런데 시스템 프롬프트에 작업 설명도 넣고, 퓨샷 예시도 넣고, 사용자 프롬프트에 블로그 본문까지 담다 보면 자연스럽게 이런 걱정이 생겨요. "이거 다 넣어도 되나? 한계가 있지 않을까?" 맞아요, 바로 **컨텍스트 길이** 문제예요.

## 컨텍스트 길이와 컨텍스트 효율성

프롬프트를 얼마나 넣을 수 있을까요?

답은 **모델의 최대 컨텍스트 길이**에 달려 있습니다.

### 컨텍스트 길이의 성장


| 모델             | 컨텍스트 길이     | 처리 가능한 분량         |
| -------------- | ----------- | ----------------- |
| GPT-2 (2019)   | 1K 토큰       | 짧은 글 한 편          |
| GPT-3 (2020)   | 2K 토큰       | 대학 과제 수준          |
| GPT-3.5        | 4K → 16K 토큰 | 법률 문서 한 장         |
| GPT-4          | 128K 토큰     | 중간 크기 책           |
| Claude 3       | 200K 토큰     | 이 책 전체(약 160K 토큰) |
| Gemini 1.5 Pro | 2M 토큰       | 위키피디아 약 2,000페이지  |


5년 만에 1K에서 2M으로 2,000배 증가했습니다.

### 컨텍스트가 길다고 다 좋은 건 아니다: Lost in the Middle

프롬프트가 길어질수록 모델이 모든 내용을 동등하게 잘 처리할까요? 안타깝게도 그렇지 않습니다.

연구에 따르면 모델은 **프롬프트의 시작과 끝에 있는 내용은 잘 처리하지만, 중간에 있는 내용은 놓치는 경향**이 있습니다([Liu et al., 2023](https://arxiv.org/abs/2005.14165)). 이를 **"Lost in the Middle"** 문제라고 부릅니다.

```
프롬프트 위치에 따른 모델 성능 (개념도)

높음 ▲
     |  *               *
     |    *           *
     |      *       *
     |        *   *
낮음 |          *
     +---------------------------→
     앞         중간        끝
```

### 건초더미 속 바늘 (Needle in a Haystack) 테스트

이 문제를 측정하는 표준 방법입니다. 무작위 정보("바늘")를 긴 프롬프트("건초더미")의 여러 위치에 삽입하고, 모델에게 그 정보를 찾도록 요청합니다.

```
[예시]
긴 JSON 데이터 속에 특정 키-값 쌍을 숨겨두고...

{"a3f9b2c1": "d4e5f6a7",
 "b8c9d0e1": "f2a3b4c5",   ← 이게 바늘
 "c1d2e3f4": "a6b7c8d9", ...}

Key: "b8c9d0e1"
Corresponding value:        ← 모델이 정확히 찾아내는지 확인
```

바늘의 위치를 앞/중간/끝으로 이동해가며 정답률을 측정합니다. 테스트할 때는 모델의 학습 데이터에 없을 만한 **정보**를 바늘로 씁니다.

우리 프로젝트의 `content_text`가 아주 긴 블로그 글이라면, 전체를 그대로 넣으면 중간 내용을 놓칠 수 있어요. 

따라서, 건초더미 속 바늘을 실습 때 진행해볼게요.

## 명확하고 명시적인 지시 작성하기

모호한 프롬프트는 모델이 형식을 스스로 정하게 만듭니다.

```python
# ❌ 모호 — 모델이 형식을 스스로 정함
("system", "블로그 글을 요약해 줘.")

# ✅ 명확 — 형식, 키, 출력 규칙까지 명시
("system", """블로그 글을 읽고 아래 JSON 형식으로만 응답하세요.
다른 텍스트 없이 순수 JSON만 반환하세요.

{
  "summary": "2~3문장 핵심 요약",
  "key_points": ["포인트 1", "포인트 2", "포인트 3"]
}""")
```

원하지 않는 동작이 나오면 그 자리에서 프롬프트에 명시적으로 금지합니다. 모델이 마크다운 코드블록으로 감싸서 응답한다면 "코드블록 없이 순수 JSON만 반환하세요"를 추가하면 됩니다.

페르소나도 여기에 해당합니다. 시스템 프롬프트에 역할 한 줄을 추가하면 응답의 톤과 관점이 달라집니다.

```python
# 페르소나 없음
("system", "블로그 글을 JSON으로 요약하세요.")
→ 평이한 요약, 장르 구분 없이 동일한 톤

# 페르소나 부여
("system", "당신은 개발자를 위한 기술 블로그 요약 전문가입니다. 블로그 글을 JSON으로 요약하세요.")
→ 기술적 맥락을 살린 요약, 핵심 개념 위주로 응답
```

---

## 충분한 컨텍스트 제공하기

**컨텍스트가 없으면 모델은 내부 지식으로 메워버립니다.** 이게 환각(hallucination)의 가장 흔한 원인입니다.

```python
# ❌ 컨텍스트 없음 → 제목만 보고 내용을 지어냄
("user", "제목: {title}")

# ✅ 컨텍스트 있음 → 실제 글 내용을 바탕으로 요약
("user", "제목: {title}\n\n내용:\n{content_text}")
```

컨텍스트를 직접 넣는 방식 외에, 모델이 필요한 정보를 스스로 검색하게 하는 RAG 파이프라인 같은 방법도 있습니다. 이는 6장에서 다룹니다.

---

## 모델에게 생각할 시간 주기 (Chain-of-Thought)

"단계별로 생각하세요"라고 한 줄 추가하는 것만으로 모델의 추론이 달라집니다. 환각도 줄어드는 효과가 있습니다.

```python
("system", """블로그 글을 읽고 아래 순서로 생각한 뒤 JSON으로 응답하세요.
1. 글의 핵심 주제를 한 문장으로 파악합니다.
2. 독자가 알아야 할 핵심 포인트 3개를 고릅니다.
3. 1, 2를 바탕으로 summary와 key_points를 작성합니다.

최종 출력은 JSON만 반환하세요.""")
```

추론 과정을 응답에 포함시키지 않으려면 "생각 과정은 출력하지 마세요"를 명시합니다. 최신 모델(Claude, o1 등)은 이 추론을 내부에서 처리하는 **extended thinking** 기능을 제공하기도 합니다.

단, 중간 추론 단계가 추가되는 만큼 **지연 시간과 비용이 늘어납니다.** 품질과 속도 사이의 트레이드오프입니다.

| **사용자 프롬프트**           | 실제 작업. 요청마다 달라지는 입력          | `("user", "제목: {title}...")`            |

| **컨텍스트 길이**            | 프롬프트에 담을 수 있는 최대 토큰 수        | `content_text`가 너무 길면 잘라서 넣는 것 고려       |
| **Lost in the Middle** | 프롬프트 중간 내용은 모델이 놓치기 쉬움       | 중요한 지시는 앞이나 끝에 배치                       |
| **강건성**                | 프롬프트 변화에 얼마나 안정적인지           | 더 강한 모델일수록 높음                           |
